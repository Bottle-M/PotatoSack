# PotatoSack worlds→paths 迁移问题清单

> 审查日期：2026-06-18
>
> 审查范围：因 Minecraft Java Edition 26.1 改动了世界/存档目录结构（level.dat 不再能凭世界名定位到所在目录），插件把"按世界名配置备份目标（`worlds`）"改为"按目录路径配置（`paths`）"。本次审查工作区中尚未提交的改动是否改全、业务逻辑有无遗漏。
>
> 审查方法：直接读码 + 对抗式多 agent workflow（8 个候选发现逐一独立验证、4 个维度搜寻新问题，13 个 agent）。8 个候选发现全部 high-confidence 确认，无一被反驳；另新增 6 个问题。

## 结论摘要

- **API 重命名基本改全**：全仓已无 `WORLDS` / `WorldRecord` / `getWorldRecord` / `getWorldRecordsFile` / `writeWorldRecord` 等旧符号的**活引用**。
- 但有 **死代码**（`resolveWorld` 及两个 import）和一批**陈旧注释**残留。
- 业务逻辑存在 **2 个 critical**（其中 1 个是 workflow 新挖出、上轮初查未覆盖的数据丢失漏洞），需优先修复。

## 🔴 Critical

### C1 — 全量备份对嵌套路径产出空/损坏包

**现象**：当 `paths` 配置了服务端根目录下的**非顶层**路径（如示例里的 `plugins/GroupManager`、`/workspace/server/world`）时，全量备份 `full.zip` 里这些路径下的文件**几乎全被跳过**，得到空/不完整的包；而 `_备份路径标识.json` 哈希记录却正确写入了这些文件，造成"记录说有、包里没有"的自相矛盾。流式压缩上传**不是**规避手段。

**根因**：`Utils.scanPeerDirsToZipPaths`（`Utils.java:471`）以 `srcDir.getName()`（仅最后一段）作为 zip 内路径前缀调用 `dirFilesToZipFilePaths`。对 `plugins/GroupManager`：

1. `getName()` = `"GroupManager"`，丢了 `plugins/`。
2. `dirFilesToZipFilePaths`（`Utils.java:442`）拼出 `currentDirOrFilePath = "GroupManager/users.json"`。
3. `new ZipFilePath(Utils.pathAbsToServer("GroupManager/users.json"), "GroupManager/users.json")`（`Utils.java:448-451`）。
4. `pathAbsToServer`（`Utils.java:220-225`）= `new File(worldContainerDir, "GroupManager/users.json")` → `<serverRoot>/GroupManager/users.json`（**错误**，真实文件在 `<serverRoot>/plugins/GroupManager/users.json`；`pathAbsToServer` 不调 `toRealPath`，无人补救）。
5. `zipSpecificFilesUtil`（`Utils.java:355-358`）发现 `new File(filePath)` 不存在 → 打印 `"(Unexpected!) File ... not found while compressing, ... ignored"` 并 `continue` 跳过。

旧代码不暴露此问题，是因为旧的 worlds 都是服务端根目录下的**顶层**目录，`getName()` 恰好等于完整的服务端根相对路径。改成任意 paths 后该假设失效。

**影响范围**：
- 传统上传与流式上传**都**走 `Utils.zipSpecificFilesUtil`（流式经 `OneDriveStreamedZipUploader.java:347,380`），流式无法绕过。
- 示例配置 `configs.yml:68-71` 直接列了 `plugins/GroupManager` 与 `/workspace/server/world`，构造函数（`BackupMaker.java:75-90`）只校验"真实目录 + 不交叠"，不拒绝嵌套路径，故可直达 `makeFullBackup`（`BackupMaker.java:468`）。
- 顶层路径（`world`、`world_nether`）不受影响。

**修复**：`Utils.java:471` 把 `srcDir.getName()` 改为 `Utils.pathRelativeToServer(srcDir)`（对服务端根返回 `""`，抛出的 IOException 已被外层 catch 覆盖）。`dirFilesToZipFilePaths:442` 已正确处理 `parentDirPath==""`。这一改同时修复 C1、H1、以及服务端根（`__root`）双前缀的情况。建议补一个嵌套路径的回归测试。

### C2 — 增量备份在上传成功前就写新哈希记录，失败后静默丢变更

**现象**：增量备份若在压缩/上传阶段失败，**下次**增量会把这些从未真正上传的变更文件**永久跳过**，造成静默数据丢失。

**根因**：`makeIncreBackup` 在逐路径循环里（`BackupMaker.java:590-615`）先 `getDirFileRecords` 取旧哈希、算 diff、收集 `increFilePaths`，然后**立刻** `writeDirFileRecords(backupConfPath, currentFileHashes)`（`BackupMaker.java:614`）把新哈希落盘——**早于**压缩（`672`）和上传（`675`）。若 `zipSpecificFiles` 返回 false（`672`）或 `uploadFile` 失败（`675-677`）`return false`，记录已被覆盖成"当前哈希"。`BackupChecker`（`71-81`）catch 住异常仅记日志；下次增量把这些新哈希当"上次基准"，而磁盘文件未变 → 无 diff → 上次没上传成功的变更文件被跳过。

迁移还放大了它：`getCurrentFileHashes` 现在声明 `throws IOException`（`Utils.java:235`，因 `pathRelativeToServer` 用 `toRealPath`，`Utils.java:179`），扫描中途（`592`）对"在 `listFiles()` 与 `toRealPath()` 之间被删除"的文件会抛 `NoSuchFileException`，导致前几个路径记录已写、其 `increFilePaths` 却未上传的半写状态。

> 注：写记录的时机是**既有逻辑**（旧 `writeWorldRecord` 也在同位置），非本次迁移引入；但属备份正确性硬伤，且迁移让它更易触发，建议一并修。

**修复**：把 `writeDirFileRecords`（`BackupMaker.java:614`）移到上传成功之后（`677` 之后，与 `backup.json` 更新一起），或写临时文件、上传成功后才提升（失败回滚）。同时让 `getCurrentFileHashes` 对瞬时缺失的文件跳过而非整体中止，避免中途抛出留下半写状态。

## 🟠 High

### H1 — 全量/增量 zip 内 entry 命名不一致

**现象**：同一个文件在全量包和增量包里的 entry 名不同。全量用 `srcDir.getName()` 派生的截短名（`GroupManager/users.yml`），增量用完整的服务端根相对 key（`plugins/GroupManager/users.yml`，`BackupMaker.java:605-611`，key 来自 `getCurrentFileHashes`→`pathRelativeToServer`，`Utils.java:244`）。`deleted.files`（`BackupMaker.java:597-599,625`）也用完整 key，和全量包对不上。

**影响**：插件**无 restore/解压逻辑**（唯一命令是 `/potatosack reload`；`Client.downloadFile` 只用于拉记录 JSON，`BackupMaker.java:412`），所以当前无插件内还原后果；但任何外部/手动按 entry 路径把增量包叠加到全量包的还原都会失败，也是未来 restore 命令的隐患。

**修复**：修 C1 后全量 entry 名即等于完整服务端根相对路径，与增量一致，本问题自动消除。

## 🟡 Medium

### M1 — `backupConfPathToNormalizedName` 非单射，记录文件名可能撞车

**现象**：两个不同的备份路径可能归一化到同一个 `_xxx.json` 哈希记录文件名，导致增量变更检测串台。

**根因**：`backupConfPathToNormalizedName`（`BackupMaker.java:167-178`）第 177 行 `normalizedPath.replace("/", "__").replaceAll("[^\\p{L}\\p{N}._-]", "_")` 非单射：
- `a/b` → `a__b`；单段目录 `a__b` → `a__b`（撞）。
- `a b`（空格）→ `a_b`；`a_b` → `a_b`（撞）。

`getDirFileRecordsFile`（`371-374`）、`getDirFileRecords`（`331-353`，含按归一化名从云端拉取）、`writeDirFileRecords`（`275-292`，**全量替换不合并**）、上传循环（`535-539,704-708`）都以该归一化名为键。第二个撞名路径会覆盖第一个的"上次基准"，下次增量对第一个路径用错的基准算 diff。启动时的唯一校验 `hasOverlappingRelativePaths`（`Utils.java:199-211`）只查父子/相等，查不出这种撞名。示例里的 `plugins/GroupManager` → `plugins__GroupManager` 可能与一个名为 `plugins__GroupManager` 的顶层目录撞名。

**修复**：使映射单射（如先把段内字面 `__` 转义再替换 `/`→`__`，或用不可能出现在文件名中的分隔符+长度前缀）；或在构造函数（`BackupMaker.java:92` 旁）加归一化名冲突检查，撞名即拒绝配置。

### M2 — `disableWorldsSave(plugin, null)` 关掉全部世界自动保存

**现象**：流式备份分支现在对**所有**已加载世界关闭自动保存，包括未被备份的世界。

**根因**：`makeFullBackup`（`BackupMaker.java:477`）与 `makeIncreBackup`（`649`）流式分支改为 `Utils.disableWorldsSave(plugin, null)`（旧代码是 `disableWorldsSave(plugin, worlds)` 定向列表）。`null` → `Bukkit.getWorlds()`（`Utils.java:547-549`）= 全部世界。当 `paths` 只含非世界目录（如 `plugins/GroupManager`）时，仍会暂停所有世界保存，造成不必要的全局卡顿，也可能干扰刻意把某世界 `autoSave=false` 的其它插件。TODO（`476,648`）已知，待结合 `.potatosackignore` 处理。恢复逻辑（`enableWorldsSave`，`486-528`）是对称的，不会卡死。传统（非流式）分支不调 `disableWorldsSave`，故仅流式分支受影响。

**修复**：只关其 world 文件夹（解析为真实路径）落在某个配置路径内的世界，把该世界名列表传给 `disableWorldsSave`；若无世界命中则跳过。

### M3 — 相对路径解析基准不一致（`user.dir` vs `worldContainerDir`）

**现象**：相对配置路径在"扫描/打包"与"key 生成/重构"两处按不同基准解析，标准启动下两者重合故潜伏，但 `user.dir != worldContainerDir` 时会哈希/读取位置错位。

**根因**：
- 扫描/打包用 `new File(path)`，按 JVM `user.dir` 解析：`BackupMaker.java:458,592`、`Utils.java:309`（`getTimeSnapshotForAllFilesUnderPaths`）、`Utils.java:469`（`scanPeerDirsToZipPaths`）。
- key 生成/重构用 `worldContainerDir`：`Utils.java:173,222`；`worldContainerDir = getServer().getWorldContainer()`（`PotatoSack.java:39`）。

迁移后 `paths` 是任意相对串（而非 Bukkit 解析的世界文件夹），使该不一致成为真实隐患。

**修复**：新增 `Utils.resolveConfPath(String)`，相对路径统一按 `new File(PotatoSack.worldContainerDir, path)` 解析（与 `pathRelativeToServer` 自身在 `Utils.java:176-178` 的解析一致），在上述四处替换裸 `new File(path)`。

---

### M4 — 空 `paths: []` 被接受，产出空的全量备份

**现象**：默认配置就是 `paths: [ ]`。空 list 时构造函数不校验，`makeFullBackup` 照常：扫描循环（`456-461`）空转、`scanPeerDirsToZipPaths` 返回空数组、`zipSpecificFiles` 产出合法的空 zip 并上传，每个全量周期在云端建一个空备份组、占用保留槽。`makeIncreBackup` 会 short-circuit（`618-621`），故只浪费全量。配置注释（`configs.yml:73`）写"留空插件无法工作"，但实际无任何拦截。

**修复**：构造函数或 `onEnable` 拒绝空/null `paths`，抛明确错误并禁用插件；`inspectConfig` 也可把空 list 视为需用户处理而非静默落盘。

## 🟢 Low

### L1 — `resolveWorld` 死代码 + 未用 import + 误导性 javadoc

`BackupMaker.java:126-157` 的 `resolveWorld(String)`（用 `NamespacedKey` 解析世界名）全仓无调用点（grep 仅命中定义）。迁移走纯 paths 路线后该世界名解析逻辑成孤儿。连带 `import org.bukkit.NamespacedKey`（13）、`import org.bukkit.World`（14）未用（`Bukkit` 在 522/691 仍用于 `getOnlinePlayers()`，保留）。javadoc（`127-135`）仍写"26.1 及其之后，还支持 Namespaced Key"，但根本没接进去，误导。

**修复**：删除 `resolveWorld` 方法与上述两个 import，保留 `Bukkit`。

### L2 — 陈旧注释仍写"世界名/世界目录/同级目录"

- `DirFileRecords.java:12,15` 字段注释仍写"世界名.json"/"世界数据目录"（类 javadoc 第 8 行已改）。
- `Utils.java:474` 报错信息仍写 `Transformation of world file paths to zip file paths failed`。
- `Utils.java:459-463` javadoc 仍说"同级目录"、`@param` 说"同一目录下的子目录"（旧假设，与新示例的任意/嵌套路径矛盾）。
- `BackupMaker.java:454,457,582,588,613` 仍写"扫描世界目录/获得世界名/世界数据目录"（459 已改，不算）。

**修复**：统一改为"备份目录/路径"术语，去掉"同级目录"假设。无行为影响。

### L3 — 无 `worlds`→`paths` 自动迁移（有意为之，已确认不误伤新装）

`inspectConfig`（`Config.java:373-428`）对缺 `version` 的旧配置设为 `"legacy"`；`PotatoSack.onEnable`（`60-66`）见 `legacy`/null 即禁用插件并提示手动升级。无任何把旧 `worlds:` 迁到 `paths:` 的代码。已确认：
- **新装不被误伤**：`configs.yml:2` 为 `version: '${project.version}'`，`pom.xml` 对 `src/main/resources` 开了 `<filtering>true</filtering>`，构建时替换为真实版本，`inspectConfig` 不会置 `legacy`，gate 不触发。
- **老用户**需手动把 `worlds:` 改 `paths:`；顶层世界名（`world`、`world_nether`）当相对路径用仍能正确定位（`pathRelativeToServer` 按 `worldContainerDir` 解析，`Utils.java:176-179`；`scanPeerDirsToZipPaths` 对 `new File("world")` 的 `getName()` 仍为 `world`，旧布局下正确）。但用户必须同时给出有效 `version` 键，否则插件保持禁用。

**修复（可选）**：加一次性迁移——当存在旧 `worlds:` 且无 `paths:` 时，把世界名写成顶层相对路径写入 `paths:` 并把 `version` 设为当前版本，而非置 `legacy` + 手动改。否则维持现状（有意设计）。

### L4 — `waitForAsyncSavesComplete` 抛 IOException 时跳过指数退避

`waitForAsyncSavesComplete`（`BackupMaker.java:750`）现在声明 `throws InterruptedException, IOException`（因 `getTimeSnapshotForAllFilesUnderPaths`/`updateFilesModificationSnapshot` 现在 throws IOException，根因是 `pathRelativeToServer` 用 `toRealPath`，`Utils.java:179`）。流式分支（`479-491`、`651-663`）中它在 `try { ... } finally { enableWorldsSave }` 内：若抛 IOException，`finally` 正确恢复自动保存（无泄漏），但 `putOffFullBackup`（`484`）/`putOffIncreBackup`（`656`）**不触发**，指数退避被跳过。异常上抛到 `BackupChecker.run`（`71-81`）仅记日志，下次 cron 直接重试无退避。旧代码 `pathRelativeToServer` 用 `Path.of(...toURI())+relativize()` 不抛 IOException，故该泄漏为迁移引入。

**修复**：在流式 try 内 catch 该 IOException 后调用 `putOffFullBackup`/`putOffIncreBackup`，或将 putOff 放入 finally/catch；可顺带让 `getCurrentFileHashes`/`pathRelativeToServer` 对瞬时缺失文件跳过而非中止。

### L5 — `hasOverlappingRelativePaths` 原地 sort 入参 list

`Utils.hasOverlappingRelativePaths`（`Utils.java:201`）`Collections.sort(relativePaths)` 直接排序入参 list。当前调用方（`BackupMaker.java:92`）传的是临时 list 且之后不再使用，暂无碍；属潜在 API 隐患。

**修复**：方法内 sort 防御性拷贝：`List<String> sorted = new ArrayList<>(relativePaths); Collections.sort(sorted);` 后遍历 `sorted`。

### L6 — 构造函数校验顺序导致报错晦涩，且漏接 `IllegalArgumentException`

`BackupMaker.java:81` 先调 `pathRelativeToServer(new File(backupConfPath))`，**再**于 `84-86` 判 `exists()/isDirectory()`。`pathRelativeToServer` 内 `toRealPath`（`Utils.java:179`）对不存在路径抛 `NoSuchFileException`，于是"路径不存在"时 `87` 行 catch 包出来的是底层 OS/Java 的 "no such file" 信息，而 `85` 行那句清晰的 `"does not exist or is not a directory."` 对该常见场景成了死代码。另外 `catch (IOException)`（`87`）漏接 `pathRelativeToServer` 对"路径在服务端根之外"抛的 `IllegalArgumentException`（`Utils.java:182`），该异常会绕过"Backup path ... is invalid:"包装直接上抛（最终被 `onEnable` 的 `catch(Exception)` 兜住，插件仍干净禁用，但诊断不清晰）。

**修复**：调整为先 `exists()/isDirectory()` 再 `pathRelativeToServer`；catch 扩到 `Exception`（或额外接 `IllegalArgumentException`），让两种情况都得到"Backup path ... is invalid: ..."包装。

## 已确认无问题 / 无需改动的点

- **新装配置不会因 version-gate 被误禁用**：Maven 资源过滤会把 `${project.version}` 替换为真实版本（见 L3）。
- **API 重命名已改全**：无活引用旧符号；仅剩 L1 死代码与 L2 陈旧注释。
- **云端记录文件名一致性**：上传（`buildDirFileRecordsFilename`）与拉取（`pullRecordsFile(buildDirFileRecordsFilename(backupConfPathToNormalizedName(...)))`）用的是同一归一化名，本地名与云端名一致。
- **`disableWorldsSave`/`enableWorldsSave` 的保存-恢复对称**：即便 M2 范围扩大到全部世界，也不会有世界被卡在 autosave=false。
